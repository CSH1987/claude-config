# 전역 세션 기본값 (모든 프로젝트 · 모든 머신)

이 파일은 claude-config 레포로 관리되며 `~/.claude/CLAUDE.md` 로 배포되어 모든 프로젝트의 모든 세션에 자동 로드됩니다.

## 응답 언어 (최우선)

- **항상 한국어로 답하세요.** 사용자가 영어로 묻거나 입력이 영어여도 기본 응답 언어는 한국어입니다.
- 예외: 코드·명령어·로그·식별자·고유명사·인용문 등 원문 그대로여야 하는 부분은 번역하지 않습니다. 설명·요약·제안은 한국어로.
- 사용자가 특정 답변에서 명시적으로 다른 언어를 요청하면 그때만 그 언어로 답합니다.

## 작업 강도 기본값 (최우선)

- 이 사용자의 기본 모드는 **최고 강도(xhigh) + ultracode/ultraplan** 스타일입니다. 모든 작업에 적용됩니다.
- **영구 적용되는 부분**: `effortLevel: "xhigh"` 가 `~/.claude/settings.json` 에 설정돼 있어 모델 추론 강도는 매 세션 자동으로 xhigh 입니다. (실제 설정값으로 진짜 영구화됨. Opus 4.7/4.8·Fable 5 필요)
- **자동 실행되는 부분**: `claude` 명령이 셸에서 `--settings ~/.claude/ultracode.json` 으로 오버라이드돼 있어, 새 세션은 보통 **ultracode 가 이미 켜진 상태**로 시작됩니다. 이 경우 시스템 리마인더에 `Ultracode is on` 이 표시됩니다 → **추가 조치 불필요**.
- **영구화가 불가능한 부분**: ultracode 의 *동적 워크플로 오케스트레이션* 과 ultraplan 은 Claude Code 설계상 세션 전용입니다 (settings.json·환경변수·훅으로 영구화 불가).
  - 만약 이 세션이 ultracode 가 아니라면(오버라이드 미적용 환경 등), 본격적인 작업(특히 복잡/다단계/대규모)을 시작하기 전에 먼저 이렇게 제안하세요:
    > "이 세션을 ultracode(동적 워크플로 오케스트레이션)로 올릴까요? `/effort ultracode` 를 실행하면 됩니다. effortLevel=xhigh 는 이미 영구 적용 중입니다."
  - 동의하면 사용자가 직접 `/effort ultracode` 를 실행해야 합니다 (훅·Claude 는 슬래시 명령을 대신 실행할 수 없음).
- 단순/사소한 작업은 굳이 제안하지 말고 그대로 진행하세요 (불필요한 오케스트레이션 비용 회피).

## 계획 우선 (ultraplan)

- 비자명한 작업은 코드를 건드리기 전에 먼저 계획을 세워 제시하고, 단계로 나눠 진행하세요.

## 도구 선택 단계 (실행 전)

- **기본 자세 = 최대한 적극(공격적)**: 설치된 도구는 **알아서 적극 활용**(작업에 맞는 전문 플러그인/스킬/에이전트가 있으면 기본으로 사용), 비자명 작업은 **요구사항을 먼저 확정**한 뒤 실행. **어지간한 작업이면 도구·딥인터뷰를 기본 적용**하고, 망설여지면 적극 쪽을 택하세요. **오직 진짜 사소·자명한 것(단순 사실 답변·한 줄 단답·자명한 1줄 수정)만** 생략(lightest-weight) — 그 외에는 적극.
- 비자명 작업은 deep-interview 취지로 요구사항을 확정한 뒤, **실행 전에** 설치된 플러그인·스킬·에이전트 중 그 작업에 가장 맞는 도구를 먼저 식별·명시하고 진행하세요. (단순/사소한 작업은 이 단계 생략)
- 라우팅 힌트(설치된 base): 브라우저 자동화·스크린샷 → **playwright**, 최신 라이브러리 문서 → **context7**, GitHub 이슈/PR/리포 → **github**(gh 로그인 시 토큰 자동), 보안·시크릿 점검 → **security-guidance**, 결정적 가드레일 → **hookify**, 스킬 제작·측정 → **skill-creator**, 플러그인/MCP 제작 → **plugin-dev·mcp-server-dev**, 신규 UI 코드 품질 → **frontend-design**, 오케스트레이션·검증·리뷰·딥인터뷰·랄프 → **oh-my-claudecode**, 전문 에이전트팀 설계 → **harness**. (배포 등 특화 플러그인은 base 에서 제외 — 필요할 때만 `claude plugin install` 로 그때그때 설치)
- 필요한 능력이 설치돼 있지 않으면 즉석에서 쓰려 하지 말고(플러그인·MCP·훅은 활성화 + 세션 재시작 후에야 동작), 먼저 "설치 + 재시작" 을 권고하세요.
- 대부분의 작업은 이미 깔린 base 로 커버되므로, 매번 새 플러그인을 깔기보다 **가진 것 안에서 라우팅**하는 것을 우선하세요.
- 사용자가 도구 이름을 몰라도, 요청을 받으면 **위 맵으로 가장 맞는 설치된 도구를 알아서 선택·사용**하세요. "어떤 도구 쓸까요?"를 매번 사용자에게 떠넘기지 말 것 — 명백하면 "이 작업은 X(설치됨)로 진행합니다" 한 줄 알리고 바로 진행.

## 플러그인 제안 (미설치 · 검증 게이트) — 좋은 게 있으면 추천하되, 검증된 것만

- base 로 **충분히 안 되는** 작업에서 더 적합한 플러그인이 있으면 **루틴으로 제안**하세요(사용자가 매번 플러그인을 직접 찾지 않아도 되도록). 단 설치는 활성화+세션 재시작이 필요하니 **동의 후**.
- **검증 게이트(필수) — 통과한 것만 추천**:
  - ✅ **공식**: `claude-plugins-official`(Anthropic 큐레이션 마켓) 플러그인은 신뢰 가능.
  - ✅ **검증된 3rd-party**: 활발한 유지보수(최근 커밋·이슈 대응) + 의미있는 채택(GitHub ⭐·실사용자) + 명확한 문서/마켓 등록 — 이 셋이 확인될 때만.
  - ❌ **금지**: 막 올라온·단일 작성자·⭐ 거의 없음·검증/리뷰 없음·관리 끊김. **"딱 하나 있는데 검증 안 된 것"을 괜찮다고 판단하지 말 것.**
  - **불확실하면 추천하지 않는다**(보수적 기본값). 추측으로 신뢰도를 단정 금지. 보안 민감(시크릿 접근·코드 실행)일수록 기준을 더 엄격히.
- **검증 방법**: 3rd-party 추천 전 가능하면 **github MCP**(repo stars·최근 커밋·이슈/PR 활동)로 신뢰 신호를 실제 확인하고 **근거를 함께** 제시. 확인 불가하거나 신호가 약하면 추천 보류.
- **발견(능동 탐색) — "없으면 찾는다"**: base 로 안 되는 작업이면 "후보가 없다"로 끝내지 말고 **github MCP 로 인기순(stars·최근 활동) 후보를 먼저 탐색**(예: `search_repositories sort=stars`)한 뒤 검증 게이트를 적용해 좁히세요.
- **중복·효율 분석 → 교체까지**: 찾은 후보를 **기존 base 와 비교 분석** — ① **중복**(같은 능력)이면 추천 안 함(base 유지), ② base 에 **없는 갭**을 메우면 추가 제안, ③ 기존 것보다 **명백히 더 효율·적합**하면 **교체(replace) 제안**도 가능. 교체는 영향이 크니 **근거(무엇이 왜 더 나은지)·마이그레이션 비용·리스크·충돌**을 함께 제시 + 검증 게이트를 더 엄격히 + **반드시 동의 후**. (인기 높아도 base 와 충돌하면 reject — 인기≠추천.)
- **제안 형식(한 작업당 1회)**: "이 작업엔 `<플러그인>`이 잘 맞습니다 — <왜 적합>. 신뢰근거: <공식 / ⭐N · 최근 활발 / 유지보수>. 설치할까요? (활성화 후 새 세션 필요)" 거절하면 같은 작업에서 다시 권하지 않음. **교체 제안 시**: "`<기존>` → `<새것>` 교체 권장 — <더 나은 점>. 비용·리스크: <마이그레이션·충돌>. 신뢰근거: <⭐N·활발>. 바꿀까요?"
- 우선순위: **설치된 base 라우팅 > 검증된 미설치 플러그인 제안 > (검증 안 되면) 제안 안 함**.

## 딥인터뷰 · OMC 모드 (적극 — 단 슬래시 명령·설치는 제안)

- **요구사항 확정(딥인터뷰 취지)은 최대한 적극·기본**: 비자명 작업은 물론 **조금이라도 모호하면 구현 전에 핵심 질문 1~3개로 요구사항을 먼저 확정**하고 진행하세요(진짜 사소·자명한 단답만 생략). 모호하면 추측하지 말고 **묶어서 한 번에** 물어 확정 → 재작업·헛작업 감소. 스펙이 크고 복잡하면 정식 **`/deep-interview`** 사용을 적극 제안.
- **OMC 슬래시 모드는 적극 제안(Claude 가 직접 실행 불가)**: 슬래시 명령은 사용자가 실행해야 하므로, 맞는 모드를 **선제적으로 제안**하고 안내하세요:
  - **"끝까지 완성·반드시 통과·검증까지"** 다단계 → **`/ralph`** (PRD 지속 루프 + 독립 리뷰어 검증)
  - 아이디어→코드 **전 과정 자동** → **`/autopilot`** · 독립 작업 **대량 병렬** → **`/ultrawork`**
  - **올리는 PR마다 자동 코드 리뷰**(항상 켜둠)를 원하면 → 그 레포에서 **`claude-review`** 한 번 실행 권고(GitHub Action + 구독 OAuth 토큰 방식, 슬래시 아님; 자세히는 README §7). 인증은 `claude_code_oauth_token` 입력, **토큰은 레포에 저장 금지**.
- 규칙: 요구사항 질문은 **묶어서 1회**(난사 금지), 모드 제안은 **한 작업당 1회**·거절 시 반복 금지, 단순/사소한 작업엔 인터뷰·모드 제안 모두 생략. (모드는 **oh-my-claudecode** 제공 — 미설치면 설치 안내.)

## Harness 자동화

- harness 플러그인이 활성화돼 있어야 합니다. SessionStart 훅(ensure-harness)이 자동 설치/복구합니다.
- 누락이 감지되면 사용자에게 알리고, 설치 후 진행하세요.

## 사용자가 잊더라도 (능동적 보장)

- 매 세션 시작 시 위 항목들을 사용자가 먼저 말하지 않아도 능동적으로 점검·반영하세요.
- 단, 한 세션에서 이미 제안/설정/처리했다면 반복하지 마세요.
