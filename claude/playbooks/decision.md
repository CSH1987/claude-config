# Playbook — 의사결정 (Decision)

> 핵심 한 줄: **근거와 함께 결정하고, 영구 기록한다.** 기록 안 된 결정은 다시 논쟁된다.

## 언제 쓰나
- 비자명하거나 되돌리기 어려운 선택.
- 여러 프로젝트/세션에 걸쳐 영향을 주는(cross-project) 결정.
- 나중에 "왜 이렇게 했지?"를 묻게 될 것.

## 방법
1. **맥락을 적는다** — 어떤 문제/상황이 이 결정을 부르나(1~3문장).
2. **옵션을 펼친다** — 2개 이상. 각각의 장단점/트레이드오프.
3. **결정을 명확히** — 무엇을 할지 모호함 없이. "앞으로 X 한다."
4. **근거** — 왜 이 옵션인가. 핵심 트레이드오프. (증거/링크 있으면)
5. **결과** — 무엇이 제약되고 무엇이 가능해지나. **재검토 트리거**(언제 다시 볼지).
6. **영구 기록** — `decisions/<machineId>/<YYYYMMDD-slug>.md` (아래 템플릿).

## 품질 바
- 결정이 한 문장으로 모호함 없이 말해진다.
- 대안이 최소 1개 검토됐다(왜 그게 아닌지).
- 재검토 트리거가 있다("X가 바뀌면 다시").
- PII 없이 적혔다(PRIVATE 저장소라 실값은 가능하나, 승급 대비 일반화 권장).

## 흔한 실패
- 근거 없이 결정 → 나중에 같은 논쟁 반복.
- 기록 안 함 → 세션 바뀌면 소실(이 시스템의 핵심을 안 씀).
- 되돌리기 트리거 없음 → 낡은 결정을 영원히 따름.

## 이 시스템과 연결
- 형식: `claude/memory/SCHEMA.md` §2 / 복붙: `claude/templates/decision-record.md`.
- 자동 포착: `/retro`가 세션의 결정을 `_pending/`에 제안 → `/reconcile`이 `decisions/`에 적용.
- 회상: 교차프로젝트 결정은 `recall`(decisions union 직read)로 cwd 무관 재사용.
- 일반적이고 재사용 가치가 크면 → `/promote`로 [playbooks](./README.md)에 승급.
