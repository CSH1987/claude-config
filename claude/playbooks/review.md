# Playbook — 검토 (Review / Verify)

> 핵심 한 줄: **"다 됐다" 선언 전에 적대적으로 검증한다.** "될 것 같다"는 증거가 아니다.

## 언제 쓰나
- 완료를 선언하기 직전(코드·문서·결정·산출물 무엇이든).
- 영향이 크거나 되돌리기 어려운 변경.

## 방법
1. **적대적으로 본다** — "이게 왜 틀렸을까?"를 먼저. 자기 작업을 반증하려 시도.
2. **증거로 확인** — 주장 말고 실행/테스트/재현. 출력을 직접 읽는다.
3. **다관점** — 정확성·보안·엣지케이스·단순성을 각각의 렌즈로.
4. **정직 보고** — 실패는 실패로, 건너뛴 건 건너뛰었다고. 통과는 증거와 함께 담담히.
5. **독립 검증(고위험)** — 큰 변경은 작성자 ≠ 검토자(별도 리뷰어/에이전트)로.

## 품질 바
- *신선한* 증거가 있다(이번에 실제로 돌려본 결과).
- 가드레일·엣지케이스가 점검됐다.
- 알려진 잔여 리스크가 은폐 없이 명시됐다.
- "통과"가 추측이 아니라 관측에 근거한다.

## 흔한 실패
- "should work / looks good"로 종료(증거 0).
- 테스트를 통과시키려 테스트를 삭제(자기기만).
- 작성자가 자기 작업을 자기 승인(독립성 부재).

## 이 시스템과 연결
- 변경 검토 → `/code-review`(정확성·재사용·효율). 실제 동작 확인 → `/verify`·`/run`.
- "끝까지 완성+검증" 다단계 → `/ralph`(독립 리뷰어 게이트).
- 보안 민감 → security-reviewer / security-guidance.
- 검토에서 나온 교훈은 [decision.md](./decision.md)로 기록해 재발 방지.
